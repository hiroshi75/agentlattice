# AgentLattice 改善提案 — CEO Jim の実体験から

**作成日**: 2026-03-16
**作成者**: Jim (CEO Agent)
**対象**: AgentLattice システム全体

---

## 1. コミュニケーション基盤の改善

### 1.1 ポーリングの非効率さ

**問題**: 各エージェントは毎サイクルでJSONLファイルをtailして新メッセージを確認する。メッセージがない場合（実際に50+サイクル連続で空だった）でも同じコストがかかる。

**改善案**:

- **ファイル監視（inotify/fswatch）ベースの通知**: JSONLに追記されたらエージェントに通知。ポーリング不要に
- **メッセージブローカー導入**: Redis Pub/Sub や SQLite の WAL モードで変更検知。JSONLは人間可読のログとして残しつつ、通知は別経路
- **`last_check.md` の廃止**: タイムスタンプ比較ではなく、行番号（オフセット）ベースで未読管理。時刻パースのオーバーヘッド削減

### 1.2 managementとの直接通信チャネル

**問題**: managementの発言はgeneralチャネル経由またはClaude Code UIからの直接入力。エージェント側からは区別しにくく、即時性も低い。

**改善案**:

- **`management` 専用チャネル**: `org/channels/management.jsonl` を作り、managementの指示は全てここに集約。エージェントは最優先で確認
- **双方向のリアルタイム通信**: エージェントからmanagementへの質問もこのチャネル経由で。現状は「チャネルに投稿して返事を待つ」しかない
- **通知サマリー**: management向けに「前回確認以降の全チャネルダイジェスト」を自動生成する仕組み

### 1.3 エージェント間の直接対話

**問題**: Kate→Samへの依頼もgeneralやengineeringチャネル経由。他のエージェントのノイズに埋もれる。

**改善案**:

- **DM（ダイレクトメッセージ）の積極活用**: CLAUDE.mdにDMの使い方は書いてあるが、実際にはほぼ全員がチャネルに投稿していた。DM利用を推奨するプロンプトを追加
- **スレッド機能**: 1つのトピックに関する会話をスレッド化。現状のフラットなJSONLではトピックが混在する

## 2. 初期チーム構成の改善

### 2.1 最初から用意すべきだったエージェント

| エージェント                               | 理由                                                                                                                        |
| ------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------- |
| **イノベーション/リサーチャー（Kai相当）** | 競合分析・市場調査を最初からやるべきだった。Sprint 3完了後に雇用したが、初期段階で市場データがあればPRDの質が格段に上がった |
| **ベータテスター（少なくとも1名）**        | Sprint 2-3の開発中にもフィードバックがあれば手戻りを減らせた。「APIキーが分からない」問題はもっと早く見つかったはず         |
| **フルスタック/DevOps（Leo相当）**         | CI/CD、ビルド、デプロイの専門家が最初からいれば、ビルドエラーの繰り返し（4回同じ修正）を避けられた                          |

### 2.2 ペルソナの改善

**問題**: 一部のエージェント（特にMeg, Rio）は指示待ちになりがちだった。Kate, Alex, Kaiは自律的に動いた。

**改善案**:

- **「自律性レベル」をペルソナに明示**: 「指示がなくても自分で次のタスクを見つけて着手する」をCritical Rulesに追加
- **「アイデア出し」を全ペルソナの定常業務に**: 今回は途中からmanagement指示で追加したが、最初から全エージェントの定常業務に「1サイクル1アイデア」を入れるべき
- **「暇な時間を作らない」ルール**: タスクが完了したら次のタスクを自分で探す。CEOに確認してから動くのではなく、動いてから報告

## 3. 開発ワークフローの改善

### 3.1 Git管理を最初から

**問題**: 各エージェントが自分のworkspaceで独立にコードを書き、後から統合した。Kate/Samのコードが何度も衝突し、ビルド修正を4回繰り返した。

**改善案**:

- **初日からGitリポジトリを初期化**: エージェント雇用時にリポジトリURLとブランチ戦略をCLAUDE.mdに記載
- **共有ワークスペース**: `agents/<name>/workspace/` ではなく、共通のリポジトリを全員がクローンして作業。各自のworkspaceはメモ・成果物用
- **CIの自動実行**: push時にビルド+テストが走る仕組みを初日から。今回はSprint 3完了後にようやく設定した

### 3.2 ビルド検証の自動化

**問題**: Kate/Samがコードを変更するたびにビルドが壊れ、CEOが手動で修正していた。

**改善案**:

- **pre-commitフック**: `cargo check` をコミット前に自動実行
- **エージェントのCLAUDE.mdに「コミット前にビルド確認」ルール追加**
- **ビルドボット**: 専用エージェントがmainブランチの変更を監視し、ビルド失敗時に自動で修正PRを作成

### 3.3 工数見積もりの廃止をデフォルトに

**問題**: PMペルソナ（Meg）がSP（ストーリーポイント）や工数見積もりに時間を費やしていた。AIエージェント開発では無意味。

**改善案**:

- **AgentLattice のデフォルトテンプレートから工数関連を削除**: sprint-prioritizerペルソナのSP計算ロジックを除去
- **PRDテンプレートから「工数」列を削除**: 「何を作るか」「誰が担当か」のみ
- **「即実行」をシステムレベルの原則に**: 全エージェントのCLAUDE.mdに自動挿入

## 4. システム機能の追加提案

### 4.1 エージェント稼働状況ダッシュボード

**問題**: CEOとして各エージェントの稼働状況を把握するのに、チャネルを全部読む必要があった。

**改善案**:

- **リアルタイムダッシュボード**: 各エージェントの最終活動時刻、現在のタスク、生産物一覧をWeb UIで表示
- **稼働率メトリクス**: 「メッセージ投稿なしのサイクル数」を追跡。暇なエージェントを自動検知
- **成果物トラッカー**: 各エージェントのworkspaceに生成されたファイルを一覧表示

### 4.2 コンテキスト共有の仕組み

**問題**: 各エージェントは自分のCLAUDE.mdしか読めない。他のエージェントの専門知識や進捗を知るには、チャネル経由でしか情報が伝わらない。

**改善案**:

- **共有知識ベース（`org/knowledge/`）の自動サマリー**: 新しいファイルが追加されたら全エージェントに通知
- **エージェントプロフィールの相互参照**: 「Kateはフロントエンド専門」「Samはバックエンド専門」を全エージェントが知っている状態に
- **コンテキストウィンドウの有効活用**: 長いチャネルログの全読み込みではなく、AI要約+関連メッセージのみ提供

### 4.3 managementへの定期レポート自動生成

**問題**: managementが「どこまで進んだ？」と聞くたびにCEOが手動でサマリーを作成していた。

**改善案**:

- **自動日次レポート**: 全チャネルの活動を集約し、「今日の成果」「ブロッカー」「次のアクション」をmanagement向けに自動生成
- **プロダクトの状態スナップショット**: ビルド状態、テスト結果、デプロイ状況を1ページで表示

### 4.4 エージェントの「記憶」の共有

**問題**: 各エージェントのmemory/は個別。CEOが学んだことを他のエージェントに伝えるにはチャネル投稿しかない。

**改善案**:

- **共有メモリ**: `org/memory/` に全員がアクセスできる共有記憶を追加
- **教訓の自動伝播**: CEOが「ビルドにはuse tauri::Managerが必要」と記録したら、関連エージェント（Kate, Sam）に自動通知

## 6. ユーザー（management）とのインタラクション改善

### 6.1 現状の課題

- managementの発言がどのタイミングで来るか予測できない
- managementの意図を正確に汲み取るのに数サイクルかかることがある
- 「あなたではなく他のメンバーに依頼して」のような方針転換を学習するのに時間がかかった

### 6.2 改善案

| 改善                           | 詳細                                                                                         |
| ------------------------------ | -------------------------------------------------------------------------------------------- |
| **management方針テンプレート** | 初期設定時に「私の好みのコミュニケーションスタイル」「委任の範囲」「即断即決 vs 相談」を設定 |
| **判断基準の明示**             | 「○○円以上の判断はmanagementに確認」のような閾値を設定                                       |
| **進捗の可視化**               | managementが見れるWebダッシュボード。チャネルを読む必要なし                                  |
| **ワンクリック承認**           | 「この判断を承認しますか？[Y/N]」のようなインタラクティブUI                                  |

## 7. 今回の活動で得た教訓（システム設計に反映すべき）

1. **「全員がイノベーター」は最初から設定すべき** — 1製品特化は組織の硬直化を招く
2. **ベータテスターは開発初期から必要** — 開発が進んでからでは手戻りが大きい
3. **Git管理は初日から** — 後から導入すると統合コストが膨大
4. **工数見積もりはAIエージェント開発では不要** — 見積もりの時間を実装に回せ
5. **暇なエージェントは即座に再配置** — 人件費（トークン）は常にかかっている
6. **競合分析は継続的に** — 市場は毎日変わる。定期的なインプットがないと判断を誤る
7. **多様な視点が重要** — エンジニアだけでは見えない。ベータテスター3名の多様性（一般/技術者/Z世代）が大きな価値を生んだ
8. **プロトタイプ > 完璧な計画** — Alexの「アイデア投稿と同時にプロトタイプ完成」が最も効率的だった

### 実現不可能な提案

| 提案 | 理由 |
|---|---|
| ファイル監視（inotify/fswatch）ベースの通知 | Claude Codeがバックグラウンドイベントを受信する仕組みがない |
| メッセージブローカー（Redis Pub/Sub等） | 同上。Claude CodeがSubscriberとして常駐不可 |
| ワンクリック承認UI | Claude CodeはCLI。ただしチャネル経由の承認フローで代替可能 |
